# Saar Blueprint Audit

Audit date: 2026-05-02

Scope: repository architecture and current frontend/backend flow compared against the Saar AI Video Production Factory target blueprint. This audit intentionally does not modify application source code, deployment configuration, environment files, RunPod/Cloudflare/Render settings, ComfyUI workflow JSON, or model-loading code.

## Target Blueprint

- Frontend: Next.js + Tailwind
- Backend: FastAPI
- Queue: Redis + Celery
- GPU: RunPod
- AI orchestration: ComfyUI headless
- Asset/video storage: Cloudflare R2
- Storyboard/keyframes: FLUX placeholder/interface for now
- Final video: generated by worker, uploaded to R2, displayed in frontend
- Memory: PostgreSQL structured memory now, ChromaDB later
- User flow:
  - prompt + images/videos/files
  - Direct Video or Generate Plan
  - scene-specific edits only
  - approve
  - generate video
  - display video in same chat/output area

## 1. What Matches The Blueprint

- Frontend stack matches: `apps/web` is a Next.js app using Tailwind, React Query, and a single primary production-factory screen.
- Backend stack matches: `apps/api` is a FastAPI app with Pydantic schemas, SQLAlchemy models, and API routes for jobs, assets, memory, billing, models, assurance, and prompt intelligence.
- Queue stack matches: Celery is configured with Redis as broker/backend in `apps/api/app/tasks.py`, and the worker queue is named `gpu`.
- GPU provider matches: RunPod endpoint routing exists through `apps/api/app/router.py` and `apps/api/app/runpod_client.py`.
- ComfyUI headless orchestration is represented: `workers/runpod-comfyui` starts a RunPod serverless handler, submits ComfyUI API workflows, waits over WebSocket/history, and uploads the output.
- R2 storage is represented: `apps/api/app/r2.py` presigns input uploads and output download URLs; the RunPod worker uploads generated media to Cloudflare R2-compatible S3.
- PostgreSQL structured memory exists: `MemoryItem` rows and `retrieve_structured_memory()` support user/project-scoped brand, style, subject, failure, and preference memory.
- Model routing exists: task types map to Wan T2V, Wan I2V, LTX preview, Hunyuan premium, and upscale endpoint defaults, with DB override support.
- Job lifecycle exists: `POST /api/jobs` creates a job, compiles context, estimates credits, optionally debits, dispatches Celery, submits to RunPod, polls status, stores an output `Asset`, and returns `output_url`.
- Direct Video / Generate Plan concept exists in the UI and API via `/api/intelligence/packet` route values `direct_video` and `generate_plan`.
- Scene-specific edits exist for generated plans: frontend scene/keyframe edits call `/api/intelligence/packet` with `edit_scene_id` or `edit_keyframe_id`, and the backend updates targeted plan/keyframe content.
- Storyboard/keyframe placeholder concept exists: keyframes use local placeholder image paths and `reference_images` are marked as placeholders.
- Preflight and smoke checks exist for major production risks, including secrets, auth, billing guard, RunPod config, R2 config, workflow placeholders, database, and proxy health.

## 2. What Is Missing

- The frontend does not currently start actual video generation from the main chat/output flow. It builds an intelligence packet and says the packet is ready for a future generator connection.
- Uploaded attachments are not wired into generation. The UI stores selected files in local component state and includes file metadata in packet settings, but it does not call `uploadAsset()` or pass `input_asset_id` into `/api/jobs`.
- The Direct Video route does not create a `/api/jobs` job. It only creates an optimized packet.
- The Generate Plan route approval step does not create a `/api/jobs` job. It only builds a final approved export packet.
- There is no frontend action for "generate video after approval" in the same output area.
- Completed videos are not displayed inline in the same chat/output area. Past jobs are listed in the sidebar, but there is no visible video player in `OutputPanel`.
- There is no end-to-end frontend path from prompt + image/video/file upload -> presign R2 upload -> job creation -> polling -> video playback.
- There is no ChromaDB integration or migration seam documented beyond structured PostgreSQL memory.
- FLUX is only mentioned as a future adapter/TODO. There is no formal FLUX placeholder interface, adapter contract, or provider abstraction for keyframe generation.
- ComfyUI workflow JSON files are documented as placeholders and preflight detects them as not production-ready.
- No Alembic migrations are present even though Alembic is listed as a dependency. Production DB schema management currently depends on `AUTO_CREATE_TABLES`.
- There is no payment-provider-confirmed credit purchase flow. The user-facing subscribe endpoint is mock-gated.

## 3. What Deviates

- The current product is more of an intelligence/plan/packet studio than a video production factory. It optimizes prompts, scenes, and keyframes before generation, but the core generation button is absent from the main UI.
- The frontend allows full plan editing through `PlanModal`, while the target flow says scene-specific edits only.
- The frontend shows "Direct video" but does not directly generate video; the label currently means direct optimized packet.
- The backend has two planning systems: `/api/intelligence/packet` and the assurance workflow (`/api/assurance/*`). This is useful, but it exceeds the minimal target architecture and may confuse the intended user flow.
- Billing is applied to intelligence packet generation separately from video generation. This can be valid, but it differs from the target emphasis on charging/protecting GPU video generation.
- Local demo auth, mock payments, seeded test coupons, and fallback plans are present. These are useful for local testing but must be carefully separated from production.
- The API allows many upload content types, including documents/spreadsheets/presentations, while the current generation pipeline only meaningfully sends image/video/audio assets to RunPod.
- RunPod worker uploads output to R2 itself, while the API also creates the output asset record after polling. This division is workable, but the API trusts the worker-returned public URL.

## 4. Useful Deviations To Keep

- Keep the structured memory layer in PostgreSQL. It is a good current-state substitute for vector memory and gives ChromaDB a clean later target.
- Keep model endpoint DB overrides. They allow production endpoint changes without redeploying code.
- Keep preflight checks. They encode several go-live blockers clearly.
- Keep the prompt/context compiler and prompt version table. They make generated jobs auditable and debuggable.
- Keep job events. They provide the basis for frontend progress updates and support/debug views.
- Keep smoke tests and proxy smoke tests. They verify the backend spine and Next proxy contract without cloud services.
- Keep mock RunPod mode for local development, as long as production checks keep it out of live use.
- Keep assurance concepts if they are repositioned as an optional advanced planning path rather than the main happy path.
- Keep credit ledger/refund mechanics. Automatic refund on failed generation is important for user trust.

## 5. Harmful Deviations To Fix

- Fix the misleading Direct Video path. Users expect a video job, but the UI only generates a packet.
- Remove or gate full-plan editing from the primary flow. It violates "scene-specific edits only" and can undermine continuity guarantees.
- Connect approval to actual generation. The current "approved export packet" creates a dead end.
- Wire attachments to R2 uploads and job creation. Without this, image-to-video cannot work from the UI despite being exposed as a task.
- Add inline job polling and video display in the same output area. The sidebar-only past videos section is not the target UX.
- Do not charge repeated intelligence-token costs for status-only approval or small UI-driven edits unless this is intentional and transparent.
- Ensure production cannot run with demo auth, mock payments, test coupons, placeholder workflows, or disabled user/billing enforcement.
- Replace `AUTO_CREATE_TABLES` as the production schema strategy with migrations before real deployment.

## 6. Broken Frontend/Backend Flows

- Attachment flow is broken:
  - UI accepts drag/drop files and shows attachment pills.
  - `uploadAsset()` exists in `apps/web/lib/api.ts`.
  - `uploadAsset()` is not used by `apps/web/app/page.tsx`.
  - No `input_asset_id` is sent to `/api/jobs`.
  - Result: image-to-video and video-upscale cannot be launched from the UI.

- Direct Video flow is incomplete:
  - UI route calls `/api/intelligence/packet` with `route: "direct_video"`.
  - Backend returns a refined prompt packet.
  - UI renders "Optimized prompt packet" and says no render has started.
  - Result: no RunPod job, no R2 output, no inline playback.

- Generate Plan approval flow is incomplete:
  - UI creates scenes/keyframes and allows scene/keyframe edits.
  - `approveAll()` rebuilds a final packet with `charge_credits=false`.
  - UI shows "ready for a future video generator connection."
  - Result: approval does not call `/api/jobs` and cannot produce final video.

- Past video display is incomplete:
  - UI fetches `/api/jobs`.
  - Sidebar labels completed jobs as "Video ready."
  - No video player or `output_url` link/playback is rendered in the main output panel.
  - Result: even completed backend jobs are not displayed in the intended chat/output area.

- User auth production handoff is incomplete:
  - Demo auth can issue local scoped sessions.
  - Real identity provider integration is not implemented.
  - Result: production requires external auth work before enabling user isolation safely.

- Payment flow is incomplete:
  - Mock checkout can add credits.
  - Real payment confirmation/webhook flow is not implemented.
  - Result: paid token flow is not launch-ready.

## 7. Security Risks

- Production defaults are unsafe unless environment flags are configured correctly: `USER_AUTH_ENFORCED`, `BILLING_ENFORCED`, `DEMO_AUTH_ENABLED=false`, `MOCK_PAYMENTS_ENABLED=false`, strong API/admin/callback/user secrets.
- When `API_AUTH_TOKEN` is empty, public API routes are effectively unauthenticated. That is acceptable only for local development.
- When `USER_AUTH_ENFORCED=false`, user-scoped reads can be made by providing arbitrary `user_id` values. This exposes jobs, wallets, memory, prompt packets, ledgers, uploads, and coupons in shared environments.
- Demo auth grants credits to first-time demo users and can issue sessions if enabled.
- Seeded test coupons create free-credit paths unless disabled or removed for production.
- The in-memory rate limiter is per-process only. It does not coordinate across multiple API instances and can reset on deploy/restart.
- Presigned upload protects size/content type before signing, but actual R2 upload enforcement depends on the signed `ContentType` and object key. There is no malware scanning or deeper file validation.
- Public R2 URLs are used for outputs. This is simple, but it means generated videos are publicly accessible to anyone with the URL unless R2/public access is configured differently.
- RunPod webhook only checks a shared callback token. That is fine as a minimum, but stronger request signing or RunPod-origin validation would reduce spoofing risk.
- The Next.js proxy injects API/admin bearer tokens server-side. This is useful, but admin proxy access depends on `SAAR_ADMIN_UI_KEY`; it should not be treated as final admin authentication.
- `AUTO_CREATE_TABLES=true` can hide migration drift and should not be relied on for production database changes.

## 8. Credit/Token Risks

- Intelligence packet generation debits credits even though `render_started=false`. This may be surprising if users think credits are only for video generation.
- Scene/keyframe edits can debit additional intelligence credits. A user can burn credits before any video is rendered.
- The frontend displays an estimated intelligence-token cost, but it is not the same as backend video generation cost from `/api/jobs/estimate`.
- Approval can be free only under a narrow approved/locked plan condition. Other small changes may still charge.
- If `BILLING_ENFORCED=false`, video jobs can run without credit debit, creating real RunPod cost exposure.
- If `USER_AUTH_ENFORCED=false`, users may manipulate `user_id` to access or spend against another wallet in a shared environment.
- Mock payments and test coupons can inflate balances if enabled outside local/staging.
- Credit estimates are static operating assumptions and are not tied to measured RunPod runtime, queue delay, GPU class, resolution, or workflow-specific cost.
- The UI can show fallback plans even if pricing plans are not fetched, which may present purchasable-looking options not actually configured server-side.
- There is no monthly plan/job limit enforcement even though pricing plans include `max_jobs_per_month`.

## 9. Priority Fixes

1. Wire frontend generation end-to-end:
   - Upload selected files through `uploadAsset()`.
   - Pass `input_asset_id` for image/video tasks.
   - Add a real "Generate video" action after Direct Video or approval.
   - Call `/api/jobs`, poll `/api/jobs/{id}`, and render the `output_url` video inline.

2. Align the Direct Video route:
   - Either rename it to "Optimize Prompt" until generation is connected, or make it create a video job immediately.

3. Align Generate Plan approval:
   - Convert approved packet data into `/api/jobs` options.
   - Preserve scene/keyframe context in job options or prompt version.
   - Start the worker after approval and show status/progress in the same panel.

4. Enforce scene-specific edits:
   - Remove, hide, or hard-gate broad full-plan editing from the normal path.
   - Keep only per-scene and per-keyframe targeted regeneration.

5. Complete video display:
   - Add inline completed/running/failed job states to `OutputPanel`.
   - Render the completed video using `output_url`.
   - Keep the sidebar as history, not the main result surface.

6. Formalize FLUX placeholder interface:
   - Define a keyframe generation adapter contract now.
   - Keep local placeholders as the default adapter.
   - Add fields for future FLUX provider output without changing the UI flow later.

7. Harden production gates:
   - Ensure production fails closed for demo auth, mock payments, placeholder workflows, missing user auth, missing billing enforcement, missing R2, and missing RunPod endpoints.
   - Remove or disable seeded test coupons outside local/test.

8. Clarify token policy:
   - Separate "planning/intelligence credits" from "render credits" in UI copy and ledger metadata, or make planning free until video generation is connected.
   - Show the backend estimate from `/api/jobs/estimate` before starting the render.
   - Enforce plan duration/job limits server-side.

9. Add database migrations:
   - Introduce Alembic revision files for current models.
   - Set production `AUTO_CREATE_TABLES=false`.

10. Replace workflow placeholders before GPU production:
   - Keep the current JSON files as contract placeholders only.
   - Validate real ComfyUI API graphs with the included preflight and RunPod worker contract.
